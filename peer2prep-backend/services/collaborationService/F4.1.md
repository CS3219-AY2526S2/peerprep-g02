# F4.1 Session Creation Specification

## Overview

This document describes the current F4.1 implementation for session creation in the Collaboration Service.

F4.1 covers:

- receiving a session creation request from the Matching Service after a successful match
- validating the request payload
- validating both matched users through the User Service
- selecting a question through the Question Service
- creating a unique active collaboration session
- enforcing idempotency for duplicate creation requests
- caching session data in Redis for the session lifetime
- allowing session creation to succeed even if Redis is unavailable

The implementation lives in the Collaboration Service and is currently scoped to backend session creation only. Real-time join, reconnection, room membership, and collaborative code sync are not part of F4.1.

## Requirement Mapping

### F4.1.1

The system shall receive a session creation request from the Matching Service upon a successful user match.

Implementation status: implemented

### F4.1.1.1

The session creation request shall include:

- unique identifier for User A
- unique identifier for User B
- selected difficulty level
- selected programming language
- topic

Implementation status: implemented

### F4.1.1.2

If any required field is missing or malformed, the system shall reject the request and return an error response.

Implementation status: implemented

### F4.1.1.3

The system shall ensure session creation requests are idempotent. Duplicate requests for the same matched user pair shall not result in multiple active sessions.

Implementation status: implemented

### F4.1.2

The system shall validate that both matched users exist and are active through the User Service before creating a session.

Implementation status: implemented through an internal dependency contract

### F4.1.2.1

If either user fails validation, the system shall:

- reject the session creation request
- return an appropriate error response
- not create or persist a session

Implementation status: implemented

### F4.1.2.2

If the User Service is unavailable, the system shall return a dependency error and abort session creation.

Implementation status: implemented

### F4.1.3

Upon successful validation, the system shall generate a session with a unique session identifier.

Implementation status: implemented

### F4.1.3.1

The system shall retrieve a question from the Question Service using the selected topic and difficulty level.

Implementation status: implemented through an internal dependency contract

### F4.1.3.2

The system shall persist session data with:

- session identifier
- user identifiers
- programming language
- question identifier
- session status
- timestamp of creation

Implementation status: implemented

Note: persistence is currently in-memory inside the Collaboration Service repository abstraction. The behavior is implemented, but durability across service restarts is not yet provided.

### F4.1.3.3

The system shall set the initial session status to `active`.

Implementation status: implemented

### F4.1.4

The system shall cache session data to enable efficient session validation and reconnection during the session lifetime.

Implementation status: implemented

### F4.1.4.1

The cached session data shall be stored only for a predefined duration matching the session lifetime and automatically removed after expiry.

Implementation status: implemented

### F4.1.4.2

If the cache is unavailable, session creation shall still succeed provided question retrieval succeeds.

Implementation status: implemented

## Internal API

### Endpoint

```http
POST /v1/api/sessions
```

This route is internal-only and protected with:

```http
x-internal-service-key: <internal-service-key>
```

The internal key is validated before any session creation logic is executed.

## Request Schema

### Request Body

```json
{
  "matchId": "match-123",
  "userAId": "user-a",
  "userBId": "user-b",
  "difficulty": "Medium",
  "language": "typescript",
  "topic": "arrays"
}
```

### Field Rules

- `matchId` is optional, but if provided must be a non-empty string
- `userAId` is required
- `userBId` is required
- `userAId` and `userBId` must be different
- `difficulty` must be one of `Easy`, `Medium`, or `Hard`
- `language` is required
- `topic` is required

### Validation Failure Response

```json
{
  "error": "INVALID_SESSION_REQUEST",
  "message": "difficulty must be one of: Easy, Medium, Hard."
}
```

### Active Session Conflict

This occurs when the same matched user pair already has another active session under a different idempotency key.

```json
{
  "error": "ACTIVE_SESSION_CONFLICT",
  "message": "An active collaboration already exists for this matched user pair.",
  "details": {
    "collaborationId": "existing-collaboration-id"
  }
}
```

## Idempotency Rules

The Collaboration Service prevents duplicate active session creation using a deterministic idempotency key.

### Current Idempotency Key

```text
matchId OR sorted(userAId:userBId)
+ difficulty
+ normalized language
+ normalized topic
```

### Behavior

- if the same active request is received again, the existing active session is returned
- if the same user pair already has another active session with different attributes, the request is rejected with `ACTIVE_SESSION_CONFLICT`

## Dependency Contracts

### User Service Validation Contract

The Collaboration Service validates that both matched users exist and are active by calling a dedicated internal User Service validation endpoint.

Configured through:

- `CS_API_GATEWAY_URL`
- `CS_USER_AUTH_BATCH_PATH`

### Outbound Request

```http
POST {CS_API_GATEWAY_URL}{CS_USER_AUTH_BATCH_PATH}
Content-Type: application/json
x-internal-service-key: <internal-service-key>
```

Request body:

```json
{
  "userIds": ["user-a", "user-b"]
}
```

### Expected Response

```json
{
  "data": {
    "users": [
      {
        "userId": "user-a",
        "status": "active"
      },
      {
        "userId": "user-b",
        "status": "active"
      }
    ]
  }
}
```

### Validation Rules

- both users must be returned
- both users must exist in the User Service
- both users must have status `active`
- otherwise session creation is aborted

The current configured path is:

```text
/v1/api/users/internal/validation/batch
```

## Question Service Selection Contract

The Collaboration Service calls a dedicated internal question selection endpoint.

Configured through:

- `CS_API_GATEWAY_URL`
- `CS_QUESTION_SELECTION_PATH`

### Outbound Request

```http
POST {CS_API_GATEWAY_URL}{CS_QUESTION_SELECTION_PATH}
Content-Type: application/json
x-internal-service-key: <internal-service-key>
```

Request body:

```json
{
  "topic": "arrays",
  "difficulty": "Medium",
  "userAId": "user-a",
  "userBId": "user-b"
}
```

### Accepted Response Shape A

```json
{
  "data": {
    "question": {
      "questionId": "q-123",
      "title": "Two Sum",
      "topic": "arrays",
      "difficulty": "Medium"
    }
  }
}
```

### Accepted Response Shape B

```json
{
  "body": {
    "quid": "q-123",
    "title": "Two Sum",
    "topic": "arrays",
    "difficulty": "Medium"
  }
}
```

### Selection Rules

- if a question ID is returned, the session proceeds
- if no question ID is present, session creation fails with `QUESTION_NOT_FOUND`
- if the dependency is unavailable, session creation fails with `QUESTION_SERVICE_UNAVAILABLE`

The current configured path is:

```text
/v1/api/questions/internal/select
```

## Session Data Model

The Collaboration Service currently stores the following session fields:

```ts
type CollaborationSession = {
  collaborationId: string;
  matchId?: string;
  userAId: string;
  userBId: string;
  difficulty: "Easy" | "Medium" | "Hard";
  language: string;
  topic: string;
  questionId: string;
  status: "active" | "inactive";
  createdAt: string;
};
```

## Persistence Design

The service currently uses an in-memory repository abstraction for active session persistence.

### Current Behavior

- new active sessions are stored in memory
- duplicate active requests return the same session
- active sessions are indexed by idempotency key and by matched user pair

### Current Limitation

- data is lost if the Collaboration Service restarts
- this satisfies behavioral requirements for F4.1, but not durable storage requirements for production

Recommended future improvement:

- replace in-memory storage with database-backed persistence

## Redis Cache Design

Session cache is implemented as best-effort acceleration and recovery support.

### Key Format

```text
collab:session:{collaborationId}
```

### Stored Fields

- `collaborationId`
- `matchId`
- `userAId`
- `userBId`
- `difficulty`
- `language`
- `topic`
- `questionId`
- `status`
- `createdAt`

### TTL

Controlled by:

- `CS_SESSION_TTL_MS`

### Cache Behavior

- the cache entry expires automatically after the configured TTL
- cache write failure does not fail session creation
- the API response includes `cacheWriteSucceeded` so callers can observe degraded behavior if needed

## Sequence Diagram

```text
Matching Service
    |
    | POST /v1/api/sessions
    | { matchId, userAId, userBId, difficulty, language, topic }
    v
Collaboration Service
    |
    | validate request payload
    | validate internal service key
    |
    | POST User Service validation request
    v
User Service
    |
    | return matched user statuses
    v
Collaboration Service
    |
    | POST Question Service selection request
    v
Question Service
    |
    | return selected question
    v
Collaboration Service
    |
    | create or reuse active session
    | cache session in Redis
    v
Matching Service
    |
    | receive collaboration session response
```
