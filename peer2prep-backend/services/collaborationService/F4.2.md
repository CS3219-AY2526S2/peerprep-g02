# F4.2 Joining a Session Specification

## Overview

F4.2 defines how an authenticated user joins an active collaboration session through the Collaboration Service socket layer.

The implementation supports:

- authenticated socket connections
- joining a collaboration session only if the user is assigned to it
- rejecting joins for missing, inactive, or unauthorized sessions
- allowing multiple sockets for the same assigned user
- preventing more than two distinct users from being present in the same session
- returning session state on successful join
- allowing assigned users to reconnect and rejoin active sessions

## Requirement Mapping

### F4.2.1

The system shall allow a user to join a collaboration session only if:

- the user is authenticated through a valid token
- the user is one of the two users assigned to the session
- the collaboration session exists
- the collaboration session status is active

Implementation status: implemented

### F4.2.1.1

Upon successful join, the system shall:

- add the user’s socket connection to the collaboration session room
- return the current session state required to resume collaboration

Implementation status: implemented

### F4.2.1.2

If any join condition fails, the system shall reject the join request and return an appropriate error response.

Implementation status: implemented

### F4.2.2

The system shall prevent more than two distinct assigned users from being present in the same collaboration session at any time.

Implementation status: implemented

### F4.2.2.1

The system may allow multiple socket connections for the same assigned user, provided no third distinct user is allowed to join the session.

Implementation status: implemented

### F4.2.2.2

If an unassigned or third user attempts to join the session, the system shall reject the request and not add the connection to the collaboration session room.

Implementation status: implemented

## Socket Authentication

The socket handshake requires a valid bearer token in either:

- `handshake.headers.authorization`
- `handshake.auth.token`

The Collaboration Service validates that token through:

```text
GET/POST via internal auth context dependency
{CS_API_GATEWAY_URL}{CS_USER_AUTH_CONTEXT_PATH}
```

If authentication fails, the socket connection is rejected before any session join is attempted.

## Join Event

### Client Event

```text
session:join
```

Payload:

```json
{
  "collaborationId": "collab-123"
}
```

### Success Acknowledgement

```json
{
  "ok": true,
  "state": {
    "session": {
      "collaborationId": "collab-123",
      "userAId": "user_1",
      "userBId": "user_2",
      "difficulty": "Medium",
      "language": "typescript",
      "topic": "arrays",
      "questionId": "q-123",
      "status": "active",
      "createdAt": "2026-03-21T12:00:00.000Z"
    },
    "questionId": "q-123",
    "codeSnapshot": "",
    "participants": [
      {
        "userId": "user_1",
        "status": "online",
        "connectionCount": 1
      },
      {
        "userId": "user_2",
        "status": "offline",
        "connectionCount": 0
      }
    ]
  }
}
```

### Failure Acknowledgement

```json
{
  "ok": false,
  "error": "SESSION_ACCESS_DENIED",
  "message": "Authenticated user is not assigned to this collaboration session."
}
```

## Join Failure Cases

### Authentication fails

Socket connection is rejected with:

```text
SOCKET_AUTHENTICATION_FAILED
```

### Session does not exist

Join acknowledgement:

```json
{
  "ok": false,
  "error": "SESSION_NOT_FOUND",
  "message": "Collaboration session was not found."
}
```

### Session is inactive

Join acknowledgement:

```json
{
  "ok": false,
  "error": "SESSION_INACTIVE",
  "message": "Collaboration session is not active."
}
```

### User is not assigned to the session

Join acknowledgement:

```json
{
  "ok": false,
  "error": "SESSION_ACCESS_DENIED",
  "message": "Authenticated user is not assigned to this collaboration session."
}
```

### Room already contains two distinct users

Join acknowledgement:

```json
{
  "ok": false,
  "error": "SESSION_CAPACITY_REACHED",
  "message": "Collaboration session already has two distinct users present."
}
```

### Invalid join payload

Join acknowledgement:

```json
{
  "ok": false,
  "error": "INVALID_JOIN_REQUEST",
  "message": "collaborationId is required to join a collaboration session."
}
```

## Presence Model

Presence is tracked per session and per assigned user.

Rules:

- multiple sockets for the same assigned user are allowed
- each user has a `connectionCount`
- a user is `online` if `connectionCount > 0`
- only two distinct users may be present in the session

## Presence Broadcast

On successful join or disconnect, the Collaboration Service emits:

```text
presence:updated
```

Payload:

```json
{
  "collaborationId": "collab-123",
  "participants": [
    {
      "userId": "user_1",
      "status": "online",
      "connectionCount": 2
    },
    {
      "userId": "user_2",
      "status": "offline",
      "connectionCount": 0
    }
  ]
}
```

## Reconnection Semantics

If an assigned user disconnects and reconnects while the session remains active:

- the user may re-authenticate
- the user may emit `session:join` again
- the system shall allow the rejoin
- the current session state is returned again

## State Returned on Join

The current join state includes:

- session metadata
- question ID
- current code snapshot
- current participant presence
