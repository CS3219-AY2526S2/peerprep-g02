Slide 2 — Technology Choices

Why Socket.IO, Redis, OT & RabbitMQ?

Every infrastructure choice is driven by a specific requirement of PeerPrep’s real-time collaborative editing for exactly 2 users

Socket.IO — Real-Time Transport

PeerPrep needs: bidirectional per-keystroke streaming, session-scoped broadcasts, automatic reconnection, and cross-instance delivery.

  • Rooms (socket.join, socket.to, io.to): Each session maps to a room collaboration:{id}, each user to user:{userId}. OT broadcasts go to the session room; submission:complete targets only the submitter’s user room. Two scoping levels out of the box.
  • Redis Adapter (@socket.io/redis-adapter): Every .emit() is published to a Redis Pub/Sub channel. If User A is on Instance 1 and User B on Instance 2, B still receives the broadcast transparently. No sticky sessions — any instance serves any session.
  • Auto-reconnection with backoff: Socket.IO clients automatically retry on disconnect. On reconnect, the client re-emits session:join and receives a full state snapshot (code, revision, hints, participants). This powers our 3-minute grace period — the reconnection logic we need is built into the protocol.
  • ACK callbacks: session:join returns the complete session state in one round trip via Socket.IO’s acknowledgement pattern. No separate REST endpoint needed for initial hydration.
  • Connection middleware (io.use): JWT is extracted from socket.handshake.auth and validated against the User Service before the connection is accepted. Authenticated userId is stored on socket.data and read for every subsequent event. Failed auth rejects the socket immediately.
  • Heartbeat (pingInterval: 25s, pingTimeout: 20s): Engine.io sends automatic ping frames. If no pong is received within 20s, the server-side disconnect handler fires — triggering presence update, socketCount decrement, and partner notification. Without this, a user who closes their laptop would appear “connected” indefinitely until a TCP timeout (minutes).
  • Disconnect event: On unexpected disconnection, the handler removes the socket binding from Redis, decrements socketCount, marks user as “disconnected” if it was their last socket, broadcasts user:disconnected to the partner, and starts the 3-minute grace period timer.

Redis — State Store + Pub/Sub + Locking

PeerPrep needs: shared state across stateless instances, sub-millisecond access for per-keystroke OT, cross-instance event delivery, and self-cleaning resources.

  • Lua scripting (EVAL): The most critical operation — every keystroke runs a Lua CAS script inside Redis: check revision → transform ops → update content + revision + append to history + trim to 50 ops + refresh TTL. All atomic in one call (<0.5ms). No race conditions between concurrent keystrokes from both users.
  • Hashes (HSET, HGETALL): Session metadata (userAId, userBId, topic, difficulty, language, status), per-user presence (status, socketCount, lastDisconnectTime), and socket bindings (collaborationId, userId, instanceId) — each stored as a Redis hash. Field-level reads and writes without deserializing entire objects.
  • Sets (SADD, SREM, SISMEMBER): Socket tracking (presence:{id}:sockets) and left-user tracking (left:{id}). SISMEMBER distinguishes “intentionally left” from “disconnected” in O(1). SCARD checks if everyone has disconnected.
  • Pub/Sub: Powers the @socket.io/redis-adapter for cross-instance room broadcasts. Dedicated pubClient/subClient connections (separate from the data client, since Pub/Sub mode blocks normal commands).
  • Pipelines: Batch multiple commands in a single round-trip. Session creation pipelines HSET + index SETs + PEXPIRE. OT updates pipeline content + revision + ops + TTL. Presence updates pipeline hash writes + set operations. Startup reconciliation pipelines batch reads of all socket bindings.
  • INCR (atomic increment): Powers race-safe AI hint rate limiting. Two concurrent hint requests get distinct INCR values (e.g., 1 and 2), so at most 2 succeed per user. If INCR returns >2, the counter is capped back.
  • SET NX PX (distributed lock): Ensures the 30-minute inactivity check runs on exactly one instance. No external coordination service needed.
  • PEXPIRE (TTL): Every session-scoped key gets a 2h TTL, refreshed on activity. Crashed or orphaned sessions auto-expire. Zero garbage collection.
  • SCAN (cursor-based iteration): On startup, reconcileOrphanedSockets() scans all socket:* keys to find bindings from dead instances and clean up stale presence data. Non-blocking, won’t freeze Redis.
  • keyPrefix: All data-client keys are auto-prefixed via ioredis keyPrefix option. Prevents collisions if multiple services share the same Redis instance. Adapter pub/sub clients deliberately skip the prefix.

OT (Operational Transformation) — Conflict Resolution

PeerPrep needs: consistent conflict resolution for exactly 2 users, server-authoritative state for code execution, and Redis-native persistence for stateless scaling.

  • Server is the single source of truth: Clients send operations, the server transforms them against the current document and broadcasts the result. When a user clicks “Run” or “Submit,” the server reads ot:{id}:content from Redis — guaranteed to be the canonical document. There is never a question of “which client’s version is correct?”
  • Redis-native state, not RAM-local: OT state (content + revision + 50-op history) lives entirely in Redis. Any Collab instance can apply a transform via the Lua CAS script regardless of which instance the client is connected to. The document survives instance crashes, rolling deploys, and reconnections to different instances.
  • Optimized for exactly 2 users: PeerPrep sessions are always a pair. OT’s transform function runs between at most 2 concurrent operations — effectively O(1) per keystroke. Strong consistency without the overhead of algorithms designed for 50+ editors.
  • Deterministic conflict resolution: When both users type at the same position simultaneously, the transform function adjusts positions predictably. Linear revision numbers + 50-op history make debugging straightforward: “client at revision 42, server at 44, here are the 2 ops to transform.”
  • Simple sync recovery: If a client falls >50 revisions behind, the server sends a full code:sync with the complete document. The client resets. No complex state-vector reconciliation needed.

RabbitMQ — Async Task Pipelines

PeerPrep needs: non-blocking code execution (10–60s in sandbox), reliable session creation across services, and decoupled service lifecycles.

  • Durable queues (durable: true): All 6 queues survive RabbitMQ broker restarts. Combined with persistent: true on every message, no session creation request or execution result is lost even if the broker or consuming service crashes mid-flight.
  • Manual acknowledgement (ch.ack / ch.nack): Messages are only removed from the queue after successful processing. If a Collab instance crashes while handling a session creation or execution result, the unacknowledged message returns to the queue and is redelivered to a surviving instance.
  • Dead-letter routing: The delay queue is configured with deadLetterExchange: "" and deadLetterRoutingKey: req_queue. Failed messages are published to the delay queue with a per-message expiration TTL. When the TTL expires, RabbitMQ automatically dead-letters the message back to the main queue for reprocessing. No custom scheduler needed.
  • Message headers (x-retry-count): Each retry increments x-retry-count in the message headers. The consumer reads this to calculate the next backoff delay (2s × 2^retryCount) and to know when max retries (5) have been exhausted. Retry state travels with the message, not stored externally.
  • Prefetch (prefetch: 1): Each instance processes one message at a time. Work is distributed fairly across instances. A slow session creation doesn’t starve the queue. Also prevents overload during spikes.
  • Backpressure handling (nack with requeue): If sendToQueue returns false (channel buffer full), the message is nack’d with requeue: true so the broker retains it for redelivery later. Prevents silent message loss under load.
  • Non-blocking execution pipeline: Code runs in a Piston sandbox for 10–60s. Collab publishes to exec_req_queue and returns immediately — the Node.js event loop stays free for OT transforms and Socket.IO events. The result arrives asynchronously on exec_res_queue and is broadcast to the room.

How They Work Together

These four technologies form an integrated architecture:

  • Socket.IO — real-time bidirectional delivery (transport layer)
  • OT — conflict resolution for concurrent edits (consistency layer)
  • Redis — shared brain that makes everything stateless (state layer)
  • RabbitMQ — reliable async pipelines for heavy tasks (decoupling layer)

Socket.IO’s Redis adapter uses Redis Pub/Sub for cross-instance broadcasts. OT’s Lua CAS uses Redis for atomic transforms. RabbitMQ decouples the slow paths (execution, session creation) from the fast path (real-time editing).

Slide 3 — Service Integration

Integration Boundaries: Matching ↔ Collaboration ↔ Question

Server-driven orchestration — the backend owns session creation, question retrieval, and room assignment

Session Creation Sequence Diagram

Step-by-Step Walkthrough

  1. Matching → RabbitMQ: After finding a compatible pair, the Matching Service publishes a CreateSessionRequest (containing matchId, userAId, userBId, difficulty, language, topicId) to the durable collab_create_req_queue.
  2. RabbitMQ → Collab: The Collaboration Service consumes the message (prefetch 1). It first checks for idempotency (duplicate matchId) and pair conflicts (sorted userA:userB key).
  3. Collab → User Service: HTTP POST to /users/internal/validation/batch with both user IDs. Validates that both users exist and are active. Returns validated user data.
  4. Collab → Question Service: HTTP POST to /internal/select with topic, difficulty, and both user IDs (to avoid repeated questions). Returns a questionId, title, and functionName.
  5. Collab → Redis (session): Creates the session hash at session:{id} containing all metadata (userAId, userBId, difficulty, language, topicId, questionId, status, createdAt). All keys get a 1-hour TTL.
  6. Collab → Redis (OT): Initializes the OT document: ot:{id}:content (with a language-specific code template using the functionName) and ot:{id}:revision set to 0.
  7. Collab → Redis (indexes): Creates lookup indexes: session:pair:{A:B} (prevents duplicate active sessions), session:idempotency:{key} (dedup), and user:active-session:{userId} for both users.
  8. Collab → Redis (cache): Writes a secondary session cache at collab:session:{id} (optional, best-effort).
  9. Collab → RabbitMQ: Publishes a CreateSessionResponse (containing the full session object with collaborationId) to collab_create_res_queue.
  10. RabbitMQ → Matching: The Matching Service consumes the response. On success, it extracts the collaborationId.
  11. Matching → Both Users: Emits match_success via Socket.IO to both users’ rooms, including the collaborationId so the frontend can navigate to the collaboration page and join.

Why Server-Driven Orchestration?

  • Security: If the frontend computes session IDs and fetches questions before “notifying” the backend, it becomes the primary orchestrator — this introduces significant security risks
  • Violation pattern: Client generates a sessionId, calls Question Service directly, then tells Collab “here’s my session” — nothing prevents spoofing, replaying, or hijacking
  • Our approach: Matching publishes to a durable queue → Collab creates the session → Collab fetches the question → client only joins a room the server already set up
  • The client never sees a questionId until it joins the server-created room
  • Question retrieval is tied to server-side room initialization, not client request

Handoff: Matching → Collaboration

  • RabbitMQ durable queue decouples the two services (no direct HTTP dependency)
  • Message contains: matchId, userAId, userBId, difficulty, language, topicId
  • Idempotency: Duplicate matchId returns existing session (no re-creation)
  • Pair dedup: Sorted userA:userB pair key prevents duplicate active sessions
  • Retryable failures use exponential backoff (2s → 32s, max 5 retries)
  • Non-retryable errors (invalid users, question not found) immediately publish failure → Matching emits match_error

Question Flow Detail

  • Selection: POST /internal/select with topic + difficulty + both userIds (avoids repeated questions)
  • On join: POST /internal/get fetches full details (title, description, testCases, functionName, image)
  • Question details are cached into the session Redis hash on first join (avoids repeated external calls)
  • Code execution uses cached testCases + functionName directly from Redis
Slide 4 — OT Deep Dive

OT Conflict Resolution — How It Works

Operational Transformation mechanics: transform functions, Lua CAS atomicity, and sync recovery (see Slide 2 for why OT was chosen)

Real-Time Technology: Socket.IO + OT

  • Transport: Socket.IO (WebSocket with automatic fallback to long-polling)
  • Why Socket.IO: Built-in room abstraction, auto-reconnection, binary support, and @socket.io/redis-adapter for horizontal scaling
  • CORS: Restricted to frontend origin, credentials required
  • Heartbeat: pingInterval: 25s, pingTimeout: 20s — detects dead connections within ~45s
  • Rooms: collaboration:{id} for session broadcast, user:{userId} for targeted events
  • Auth: JWT extracted from handshake → validated against User Service on every connection

Why OT over CRDT (Y.js)?

  • Centralized authority: Server is the single source of truth — OT fits naturally with our server-driven architecture
  • Two-user constraint: PeerPrep rooms are strictly 2 users; OT excels at small participant counts with strong consistency
  • RAM-local Y.Doc problem: Y.js stores the document in server RAM — both peers must land on the same instance, defeating horizontal scaling unless you add a Redis-backed persistence layer anyway
  • Our OT: Document state lives entirely in Redis, so any Collab instance can serve any session
  • Unit of sync: Character-level operations (insert/delete at position) — fine-grained, immediate feedback

OT Conflict Resolution Sequence

Step-by-Step Walkthrough

  1. Frontend → Collab: Client emits code:change with the OT operations and its current clientRevision number.
  2. Collab → Redis: Server reads the current document content, revision, and operation history from Redis (ot:{id}:content, ot:{id}:revision, ot:{id}:ops).
  3. Redis → Collab: Returns the authoritative document state and the list of operations since the client’s revision.
  4. Transform (server-side): If the client revision is behind, the server transforms the incoming operations against all intervening server operations (from other users only) using the OT algorithm. Server operations have “right” priority for tie-breaking.
  5. Apply: The transformed operations are applied to the current document content to produce the new text.
  6. Lua CAS → Redis: The server runs an atomic Lua compare-and-swap script: only writes the new content + revision + operation history if the revision hasn’t changed since step 2. If it changed (another concurrent edit landed), returns 0 and the server retries from step 2 (up to 5 times).
  7. Redis → Collab: Returns 1 (success) or 0 (conflict, retry needed).
  8. Collab → Redis: Updates the session activity timestamp (activity:{id}) to prevent inactivity timeout.
  9. Collab → Sender: Sends code:ack to the originating client with the newRevision number, confirming the change was accepted.
  10. Collab → Room: Broadcasts the transformed operations to all other sockets in the collaboration:{id} room via code:change (cross-instance via Redis adapter).

Lua CAS Script (Atomic Update)

-- Redis Lua: atomic compare-and-swap
local curRev = redis.call("GET", revKey)
if curRev ~= expectedRev then return 0 end
redis.call("SET", contentKey, newContent)
redis.call("SET", revKey, newRevision)
redis.call("LPUSH", opsKey, opJson)
redis.call("LTRIM", opsKey, 0, 49)
return 1

Sync Recovery

  • If OT transform fails (client too far out of sync), server sends code:sync with full document snapshot
  • Client resets its local state to the server’s authoritative version
  • Operation history kept to 50 entries (LTRIM) to bound memory
Slide 5 — Session Management

Room Authorization, Presence & Multi-Tab Handling

Two-layer auth, socket-count presence tracking, and 3-minute reconnection grace period

Room Authorization (Two Layers)

  • Layer 1 — Socket Auth: On every connection, JWT is extracted from handshake and validated against User Service (/users/internal/authz/context). Invalid tokens are rejected before any room access.
  • Layer 2 — Room Membership: On session:join, server loads session from Redis and verifies userId is in [userAId, userBId]. Returns SESSION_ACCESS_DENIED if not.
  • Token expiry mid-session: Socket.IO auth runs only at connection time. Since sessions are short-lived (1h TTL), the heartbeat mechanism (ping/pong) detects dead connections, and the session’s own TTL enforces expiry.
  • Code changes also guarded: Every code:change re-checks that the userId is one of the two assigned users and the session is still active
Session Join Sequence Diagram

Step-by-Step Walkthrough

  1. [Auth] Frontend → Collab: Client opens a Socket.IO connection with JWT in the handshake headers or auth object.
  2. [Auth] Collab → User Service: The socket auth middleware calls GET /users/internal/authz/context with the JWT and x-internal-service-key. If the user is not active, the connection is rejected.
  3. Frontend → Collab: Client emits session:join with { collaborationId }.
  4. Collab → Redis: Loads the session hash (HGETALL session:{id}) and validates: session exists, is active, and userId matches either userAId or userBId.
  5. Collab → Redis: Checks if user previously left (SISMEMBER left:{id}), loads presence data, and verifies the reconnection grace period hasn’t expired (3-minute default).
  6. Collab → Question Service: Calls POST /internal/get with the questionId to fetch full question details (title, description, testCases, functionName, image).
  7. Collab → Redis: On first join, caches the question details into the session hash so future joins and code executions don’t need to call the Question Service again.
  8. Collab → Redis: Registers the socket binding (socket:{socketId} → {collaborationId, userId, instanceId}), adds socket to the session’s socket set (SADD), increments socketCount, and reads the OT document state (code + revision).
  9. Collab → Redis: Reads existing AI hints (LRANGE hints:{id}), hint count, and the last execution output (GET output:{id}).
  10. Collab → User Service: Calls POST /validation/batch to get display names for both users.
  11. Collab → Frontend: ACK callback returns the full state: session metadata, question details, current code + revision, AI hints, remaining hints count, user names, and presence info.
  12. Collab → Room: Broadcasts user:joined (to other sockets) and presence:updated (to all sockets in the room).

Active Session Check

  • Before entering matching, client emits session:check-active
  • Server checks user:active-session:{userId} index in Redis
  • If active session found → returns collaborationId for rejoin
  • Prevents users from starting a new match while an active session exists
Presence State Machine

State Transitions Explained

  • CONNECTED → CONNECTED (self-loop): Opening a new browser tab adds another Socket.IO connection. socketCount increments; status stays “connected”.
  • CONNECTED → DISCONNECTED: When the last socket closes (socketCount hits 0), status changes to “disconnected” and lastDisconnectTime is recorded. The 3-minute grace period timer starts. Partner is notified via user:disconnected.
  • DISCONNECTED → CONNECTED: If the user reconnects within the 3-minute grace period (re-emits session:join), status is restored to “connected”, disconnect time is cleared, and a full state snapshot is sent.
  • CONNECTED → LEFT: Explicit session:leave immediately marks the user as “left” (added to left:{id} set). This is permanent — rejoin is blocked.
  • DISCONNECTED → LEFT: If the grace period expires without reconnection, the user is effectively treated as gone. Rejoin returns REJOIN_GRACE_PERIOD_EXPIRED.
  • LEFT → SESSION ENDED: When both users are resolved (both left, or one left + other disconnected), the session ends: all Redis resources are cleaned up and session:ended is emitted.
  • DISCONNECTED → SESSION ENDED: If a user’s partner leaves while they are disconnected, the session stays alive until the disconnect grace period (3 min) expires. If they reconnect within grace, they find the session active with the partner marked as “left”. If they don’t reconnect, the periodic check ends the session once grace expires.
  • CONNECTED → SESSION ENDED (inactivity): If no code changes occur for 30 minutes (session-level check, not per-user), the periodic inactivity check ends the session for both users simultaneously, regardless of individual presence states.

Multi-Tab Handling

  • Each browser tab = 1 Socket.IO connection = 1 socket in the room
  • socketCount tracks how many sockets a user has in a session
  • User is only “disconnected” when the last socket closes (socketCount hits 0)
  • Closing one tab while another is open: no state change, no notification to partner
  • All tabs receive the same OT broadcasts — document stays in sync across tabs
  • Socket binding in Redis: socket:{socketId} → { collaborationId, userId, instanceId }

Disconnect / Reconnect / Rejoin

  • On disconnect: Remove socket binding, decrement socketCount. If last socket: set status = “disconnected”, record lastDisconnectTime, notify partner
  • Grace period: 3 minutes (configurable via CS_DISCONNECT_GRACE_MS)
  • On reconnect within grace: Client re-emits session:join → server restores status to “connected”, clears disconnect time, sends full state snapshot
  • Grace expired: Rejoin blocked with REJOIN_GRACE_PERIOD_EXPIRED
  • Session state preserved in Redis: OT document, execution output, hints, presence all survive disconnects (1h TTL)
  • Startup recovery: reconcileOrphanedSockets() scans for socket bindings from dead instances and cleans up stale presence data

Normal vs exceptional exit paths, resource cleanup, and crash recovery

Session End Sequence Diagram

Step-by-Step Walkthrough

  1. Trigger detected: Session end is triggered by one of three conditions: both users left, one left while the other is disconnected, or no code changes for 30 minutes (inactivity timeout detected by periodic distributed check).
  2. Collab → Redis: Reads the final code state (GET ot:{id}:content + revision) for potential use in attempt recording.
  3. Collab → Redis: Marks the session as inactive (HSET session:{id} status → "inactive").
  4. Collab → Redis: Deletes lookup indexes: session:pair:{A:B}, session:idempotency:{key}, and user:active-session:{userId} for both users — allowing them to start new sessions.
  5. Collab → Redis: Deletes all presence data: individual socket bindings (socket:{socketId}), the socket set, the left set, and per-user presence hashes.
  6. Collab → Redis: Deletes the OT document: content, revision, and operation history.
  7. Collab → Redis: Deletes execution output, AI hints list, and hint counters.
  8. Collab → Frontend: Emits session:ended to the entire room with the reason (e.g., "both_users_left", "inactivity_timeout").
  9. Collab → Frontend: Closes (disconnects) all remaining sockets in the room.

Safety net: Even if cleanup fails, all Redis keys have a 1-hour TTL and will auto-expire.

Crash Recovery Detail

  • Orphan reconciliation (reconcileOrphanedSockets): On startup, each instance scans all socket:{id} bindings in Redis. Each binding contains an instanceId. The instance batch-checks liveness keys (instance:{uuid} with 30s TTL). If the owning instance’s liveness key is missing, the socket is orphaned — its binding is deleted, socketCount decremented, and presence updated.
  • Instance heartbeat: Each instance writes instance:{uuid} with 30s TTL, refreshed every 10s. If an instance crashes, its key expires within 30s, marking all its sockets as orphans for the next reconciliation pass.
  • RabbitMQ redelivery: All queues are durable with persistent messages and prefetch: 1. When an instance crashes mid-processing, the unacknowledged message returns to the queue and is delivered to a surviving instance. Retryable failures use exponential backoff (2s → 32s, max 5 retries) via dead-letter delay queues.
  • Socket.IO auto-reconnect: Clients have built-in reconnection logic. On reconnect, they re-emit session:join — the server checks the 3-min grace period, restores presence to “connected”, and sends a full state snapshot (code, revision, participants, hints, output).
  • No sticky sessions: Since all state is in Redis and room broadcasts use the Redis adapter, clients can reconnect to any instance, not just the one they were originally on.

Graceful Shutdown (Ordered Teardown)

  • 1. Stop inactivity check interval
  • 2. Close HTTP server (reject new connections)
  • 3. Close Socket.IO (triggers disconnect handlers → cleans Redis presence)
  • 4. Close RabbitMQ (isShuttingDown flag prevents reconnect storms)
  • 5. Quit Redis adapter pub/sub clients
  • 6. Stop instance heartbeat (delete liveness key)
  • 7. Close main Redis client → exit (10s hard deadline)

Order matters: Socket.IO disconnect handlers need Redis, so Redis closes last.

Slide 7 — Execution & AI Hints

Code Execution Pipeline & AI Hint System

Asynchronous execution via RabbitMQ + Piston sandbox, and Gemini-powered contextual hints with atomic rate limiting

Code Execution Sequence Diagram

Step-by-Step Walkthrough

  1. Frontend → Collab: Client emits code:run (test only) or code:submit (test + record attempt) with the collaborationId.
  2. Collab → Room: Immediately broadcasts code:running to the entire room so both users see a loading/spinner state.
  3. Collab → Redis: Reads the current OT document (live code) and the cached test cases + function name from the session hash.
  4. Collab → Redis: Sets a safety-net key exec:pending:{correlationId} with a 65-second TTL. If no result arrives in time, this triggers an automatic timeout error to the room.
  5. Collab → RabbitMQ: Publishes an ExecutionRequestMessage to exec_req_queue containing the code, language, function name, test cases, and metadata.
  6. RabbitMQ → Execution Service: The Execution Service consumes the message, wraps the user code with test harness logic, and runs it inside a Piston sandbox (isolated container).
  7. Execution Service → RabbitMQ: Publishes the result to exec_res_queue with per-test-case pass/fail, actual vs. expected output, and execution time.
  8. RabbitMQ → Collab: The Collaboration Service consumes the execution result.
  9. Collab → Redis: Stores the output at output:{id} (so rejoining users see it) and deletes the exec:pending key.
  10. Collab → Room: Broadcasts output:updated to the entire room — both users see the execution results.
  11. [Submit only] Collab → Attempt Service: If the trigger was code:submit, an HTTP POST is made to the Attempt Service to record the code snapshot, language, test results, and session duration.
  12. [Submit only] Collab → Submitter: Emits submission:complete to the submitter’s user room only (not the whole collaboration room).

Execution Safety Measures

  • Timeout: 65s safety net via Redis key — if execution hangs, room gets a timeout error automatically
  • Sandbox: Code runs in Piston (isolated container), never on the Collab instance
  • Non-blocking: Collab publishes to RabbitMQ and returns immediately — no thread blocking
  • Durable: If Execution Service crashes, messages survive in the durable queue and are redelivered
  • Both users see results: output:updated is broadcast to the entire room
  • Submit-only persistence: Only code:submit records an attempt; code:run is ephemeral
AI Hint Sequence Diagram

Step-by-Step Walkthrough

  1. Frontend → Collab: Client emits hint:request with the collaborationId.
  2. Collab (validate): Server validates that the session exists, is active, the user is an assigned participant, and hasn’t left.
  3. Collab → Redis (pre-check): Reads hints:count:{id}:{userId} to check if the user has already used their 2-hint limit. This avoids wasting a Gemini API call if already at the cap.
  4. Collab → Redis (context): Reads the current OT document (live code), question details from the session hash, and all previous hints from hints:{id} to build the prompt context.
  5. Collab → Gemini 2.5 Flash: Sends a structured prompt including the question, difficulty, language, current code, and previous hints. The prompt instructs Gemini to give 2–3 specific sentences without revealing the full solution.
  6. Gemini → Collab: Returns the hint text (with a 15-second timeout; config: temperature=0.4, maxOutputTokens=512).
  7. Collab → Redis (atomic INCR): Atomically increments hints:count:{id}:{userId}. If the result exceeds 2, the counter is capped back to 2 and the hint is rejected — this handles race conditions where two concurrent requests both pass the pre-check.
  8. Collab → Redis (store): Appends the hint to hints:{id} list (RPUSH) with the user ID and timestamp. Refreshes the TTL.
  9. Collab → Requester: ACK callback returns the full hints array and remaining hint count for this user.
  10. Collab → Room: Broadcasts hint:updated to the entire room so both users see the new hint.

Rate Limiting (Atomic, Race-Safe)

  • Limit: 2 hints per user per session (each user gets their own counter)
  • Two-layer enforcement:
  • Pre-check (cheap): Read counter before calling Gemini — avoids wasting an API call if already at limit
  • Atomic post-check: Redis INCR on hints:count:{id}:{userId} — if result > 2, cap counter back to 2 and reject
  • Race safety: Two concurrent requests get distinct INCR values (e.g., 1 and 2), so at most 2 succeed
  • Counter + list keys expire with session TTL (1h)

Gemini Prompt Design

  • Role: “Coding tutor for a LeetCode-style challenge”
  • Context sent: question title, description, difficulty, language, current live code, all previous hints
  • If code exists: Analyze for bugs, missing steps, logical errors; reference specific parts of their code
  • If no code yet: Suggest the key algorithm/data structure and explain why
  • Constraints: 2–3 direct sentences, no generic advice, no full solutions, never repeat previous hints
  • Config: temperature=0.4, maxOutputTokens=512

How Both Peers See the Same Document

  • The problem: Without the adapter, socket.to(room).emit() only reaches sockets on the same process. Client B on Instance B would never receive Client A’s changes.
  • The solution: @socket.io/redis-adapter replaces Socket.IO’s default in-memory adapter. Every .emit() to a room is also PUBLISHed to a socket.io# Redis channel. All other instances’ subClients receive it and deliver to their local sockets in that room.
  • Two dedicated connections: pubClient publishes outgoing broadcasts; subClient subscribes to receive other instances’ broadcasts. These are separate from the main data client because once a Redis connection enters Pub/Sub mode, it can no longer run normal commands (GET, SET, EVAL).
  • Transparent to app code: The socket handler just calls socket.to(room).emit("code:change", transformedOps) — the adapter intercepts it and handles cross-instance delivery automatically. No custom redis.publish() calls anywhere in the codebase.
  • No sticky sessions needed: Any instance can serve any session since all state lives in Redis and room broadcasts reach all instances via pub/sub. Users can reconnect to any instance after a disconnect.

Distributed Coordination

  • Inactivity check: Only one instance runs it using SET NX PX distributed lock
  • OT consistency: Lua CAS script ensures atomic document updates regardless of which instance processes the change
  • RabbitMQ prefetch 1: Fair load distribution across consuming instances

Dead-Letter Retry Sequence

Step-by-Step Walkthrough

  1. Failure detected: The Collab Service fails to process a session creation message (e.g., Question Service is temporarily down). The error is classified as retryable (not in the NON_RETRYABLE_ERROR_CODES set).
  2. Collab → Delay Queue: The message is published to collab_create_req_delay_queue with a per-message TTL calculated as 2000ms × 2^retryCount. The x-retry-count header is incremented.
  3. Delay Queue (wait): The message sits in the delay queue until its TTL expires. The delay queue is configured with deadLetterExchange: "" and deadLetterRoutingKey: collab_create_req_queue.
  4. Delay Queue → Main Queue: When the TTL expires, RabbitMQ dead-letters the message back to collab_create_req_queue for reprocessing.
  5. Main Queue → Collab: The message is redelivered to the Collab Service with the incremented retry count. The cycle repeats if processing fails again.
  6. After 5 retries: The message is discarded and a failure response is published to collab_create_res_queue, which the Matching Service consumes to emit match_error to both users.

Backoff schedule: 2s → 4s → 8s → 16s → 32s (total ~62s before final failure).

Exponential Backoff

  • Base delay: 2000ms
  • Formula: base × 2^retryCount
  • Retry 1: 2s → Retry 2: 4s → Retry 3: 8s → Retry 4: 16s → Retry 5: 32s
  • Max retries: 5 (then discard + publish failure response)
  • Prefetch count: 1 (ensures fair distribution and ordering)
  • Backpressure: if sendToQueue returns false, message is nack’d for redelivery

Non-Retryable Errors (Immediate Discard)

These are logic errors that will never succeed on retry:

ACTIVE_SESSION_CONFLICT USER_VALIDATION_FAILED QUESTION_NOT_FOUND INVALID_SESSION_REQUEST

On non-retryable error: immediately publish failure to collab_create_res_queue → Matching Service emits match_error to both users

Key Configuration

// All queues asserted as durable
await ch.assertQueue(REQ_QUEUE, { durable: true });

// All messages sent with persistence
ch.sendToQueue(queue, buffer, { persistent: true });

// Delay queue with dead-letter routing
await ch.assertQueue(DELAY_QUEUE, {
  durable: true,
  deadLetterExchange: "",
  deadLetterRoutingKey: REQ_QUEUE,
});

Why Durable Queues?

  • No message loss: Both queue declarations and messages are persistent — survive RabbitMQ broker restarts
  • Decoupling: Matching Service and Execution Service don’t need to be online at the same time as Collab
  • Idempotency: Duplicate matchId returns existing session, so redelivered messages are safe
  • Default exchange: No exchanges used — direct sendToQueue for simplicity

Slide 11 — Architecture Summary

Key Architectural Decisions

The decisions that shape the Collaboration Service and why they matter

Key Architectural Decisions
DecisionWhy It Fits PeerPrep
OT for conflict resolution Server-authoritative: the document in Redis is always the canonical version for code execution. State lives in Redis (not RAM), so any instance can transform ops — no sticky sessions. 2-user sessions mean transform cost is effectively O(1).
Redis as the shared brain Serves 3 roles simultaneously: (1) state store for 18 key types with sub-ms access for per-keystroke OT, (2) Pub/Sub backbone for Socket.IO cross-instance broadcasts, (3) distributed locks via SET NX PX. TTL on every key means zero cleanup jobs — crashed sessions self-destruct.
Socket.IO for real-time transport Room abstraction maps directly to collaboration sessions. Redis adapter enables cross-instance delivery transparently. Auto-reconnection powers our 3-minute grace period. ACK callbacks return full session state on join in one round trip.
RabbitMQ for async pipelines Code execution takes 10–60s in Piston sandbox — can’t block the event loop. Durable queues survive crashes. Dead-letter routing gives us exponential backoff retries (2s→32s, 5 max) without custom scheduling. Decouples Collab, Execution, and Matching lifecycles.
Lua CAS for atomic OT The most frequent operation (every keystroke): read revision → transform → update content → append to history. Runs atomically inside Redis with zero race window. No distributed lock overhead, no retry loops, sub-millisecond.
100% stateless instances Kill any instance — clients reconnect to another, all state survives in Redis. Deploy new versions with zero downtime. Scale horizontally by adding instances. No session affinity, no instance-specific state, no migration on restart.

Design Principles

  • Redis is the brain, instances are the hands: All truth lives in Redis. Instances are stateless workers that read state, apply logic, and write back. Kill any instance — another picks up seamlessly.
  • Async by default, sync only when forced: Code execution and session creation use RabbitMQ queues. HTTP is used only when a response must block (auth, question fetch). Attempt recording fires without blocking the user.
  • Crash recovery is automatic, not manual: Socket.IO auto-reconnects clients. Redis preserves all state. RabbitMQ redelivers unacknowledged messages. Orphaned sockets are reconciled on startup. No human intervention needed.
  • Every Redis key has a TTL: Even if all cleanup code fails, keys auto-expire in 2 hours. No leaked resources, no manual purging, no state accumulation.
  • Server is the authority: Clients never trust each other directly. All OT ops go through the server. Code execution happens server-side. Session membership is validated on every event.

What Makes This Non-Trivial?

Problems that go beyond basic CRUD:

  • Real-time conflict resolution: Two users typing simultaneously at the same position — OT + Lua CAS ensures a consistent document without blocking either user
  • Horizontal scaling with rooms: User A on Instance 1 types, User B on Instance 2 sees it in <50ms via Redis Pub/Sub adapter — no sticky sessions
  • Graceful degradation: Network drops, server crashes, service timeouts — 3-min reconnection grace, durable queues, orphan reconciliation ensure the session survives
  • Distributed coordination: Inactivity timeout check, session end cleanup, and hint rate limiting all work correctly across multiple stateless instances using Redis atomic operations
  • Async execution pipeline: Code submission → RabbitMQ → sandboxed Piston execution → result broadcast, all non-blocking with 65s safety timeout
